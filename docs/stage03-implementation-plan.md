# Stage 3: Comments and Collaboration Upgrade — Implementation Plan

This document translates **docs/stage03-comments-and-collaboration-upgrade-mini-spec.md** into a concrete engineering implementation plan. No code is written in this document.

**Constraints (must be preserved):**
- Stage 1 visibility rules (TicketVisibilityService, policy, role-based ticket access).
- Stage 2 workflow system (ticket/subtask lifecycle, progress, state machine).
- Backend is the source of truth for mentions (parse, persist, fanout from DB).
- Only one-level comment replies (no reply-to-reply).
- Mentionable users must have ticket visibility.
- Mention + comment notifications must dedupe (one notification per user per comment; mention preferred).

---

## 1. Comment Thread Data Model Changes

### 1.1 Schema

- **Table:** `ticket_comments` (Prisma model `TicketComment`).
- **New column:** `parentCommentId` — optional, nullable, type `String`. If `null`, the row is a top-level comment; if set, it is a reply to the comment with that id.
- **Foreign key:** `parentCommentId` → `ticket_comments.id`. **ON DELETE CASCADE**: when a parent comment is deleted, all replies are deleted. No SET NULL.
- **Self-relation:** Add Prisma relation `parent` (TicketComment?) and `replies` (TicketComment[]) on `TicketComment` so that `parentCommentId` points to `parent` and `replies` are comments where `parentCommentId = this.id`.
- **CommentMention:** No schema change. Existing `comment_mentions` table remains.

### 1.2 Indexes

- **Index on `parentCommentId`:** Required for efficient "replies of this comment" queries when building thread shape. Name suggestion: `ticket_comments_parentCommentId_idx` (or Prisma `@@index([parentCommentId])`).
- **Deterministic ordering:** Comments are ordered by `createdAt` ascending. For thread response, use a **tiebreaker** so ordering is stable when `createdAt` is equal: e.g. `orderBy: [{ createdAt: 'asc' }, { id: 'asc' }]`. Apply this wherever comment lists are queried (findByTicket, ticket findById include).

### 1.3 Migration

- **Single migration:** Add `parentCommentId` column, FK constraint, ON DELETE CASCADE, and index on `parentCommentId`. Migration name suggestion: `YYYYMMDD_stage3_comment_threads` (e.g. `20260313000000_stage3_comment_threads`).
- **Data:** Existing rows have `parentCommentId = null` (all current comments are top-level). No data backfill required.
- **Prisma:** Regenerate client after migration. Update `schema.prisma` with the new field, relation, and index.

### 1.4 Files to touch

- `apps/api/prisma/schema.prisma` — add `parentCommentId`, relation, index.
- New file: `apps/api/prisma/migrations/<timestamp>_stage3_comment_threads/migration.sql` (generated by `prisma migrate dev` or written manually).

## 2. Backend Mention Parsing and Validation

### 2.1 Mention format

- **Canonical token:** `@[Display Name](userId)` — regex or parser must extract `userId` from this pattern only. Plain `@username` is **not** parsed for notifications.
- **Uniqueness:** Parsed list of userIds must be **unique** (one row per user per comment in `comment_mentions`).

### 2.2 Services to change

- **MentionParserService** (`apps/api/src/modules/comments/mention-parser.service.ts`):
  - **Keep:** `parseStructuredMentions(body)` — returns unique userIds from body using existing regex for `@[.*](userId)`.
  - **Add:** New method that accepts **ticket context** and **optionally** a list of **allowed (mentionable) user IDs** for that ticket. Signature suggestion: `extractMentionsForTicket(body: string, allowedUserIds: string[]): Promise<string[]>` or `extractMentions(body, options?: { ticketId: string })`. When `allowedUserIds` (or ticketId for server-side resolution) is provided, **filter** parsed userIds to only those in the allowed set (or resolve allowed set from ticket visibility and intersect). Return only valid, active, **ticket-visible** user IDs.
  - **Behavior:** If strict validation is chosen: when body contains a userId not in the allowed set, either (a) **reject** with 400 and a clear message ("Cannot mention user without ticket access") or (b) **strip** invalid mentions and persist the rest. Plan **recommends** (a) so the API contract is explicit and the frontend can restrict the typeahead to mentionable users only.
  - **Deduplication:** Before returning, ensure list is unique (existing parse already avoids duplicates; keep that guarantee).

### 2.3 Call site (CommentsService)

- **CommentsService.create** (and **update**, if mention re-parsing is done on edit):
  - **Before creating comment:** Resolve **mentionable user IDs** for this ticket (see §3 — use a shared helper or TicketVisibilityService to get users who can see the ticket).
  - Call mention parser with body and **allowedUserIds** (or ticketId so parser can resolve allowed set internally). If parser is in CommentsService, inject a dependency that can return mentionable user IDs for a ticket.
  - If any parsed mention is **not** in the allowed set and policy is "reject": throw **BadRequestException** (400) with message e.g. "Cannot mention user without ticket access" and optionally list invalid userIds.
  - If "strip" policy: only persist CommentMention rows for allowed userIds; do not throw.
  - **Existing flow:** Create comment and CommentMention rows in transaction; after commit, emit COMMENT_ADDED and, if mentions exist, MENTION_IN_COMMENT with the **filtered** mentionedUserIds (same as persisted).

### 2.4 Dependency

- MentionParserService (or CommentsService) must have a way to obtain **mentionable user IDs for a ticket**. This is implemented by the **Mentionable Users** feature (§3). Either:
  - **Option A:** CommentsService calls a new **MentionableUsersService** (or TicketsService method) `getMentionableUserIds(ticketId: string): Promise<string[]>` and passes to parser; or
  - **Option B:** MentionParserService accepts `ticketId` and uses TicketVisibilityService + User/Ticket model to resolve visible users and filter. Option A keeps visibility logic in one place (Tickets or a dedicated service) and avoids circular deps; **recommended**.

### 2.5 Files to touch

- `apps/api/src/modules/comments/mention-parser.service.ts` — add ticket-scoped extraction and optional validation (reject/strip).
- `apps/api/src/modules/comments/comments.service.ts` — before create (and update if needed), resolve mentionable user IDs; call parser with allowed set; reject or strip invalid mentions; pass only allowed userIds to CommentMention create and to domain events.
- New service or method for "mentionable user IDs for ticket" (see §3); CommentsService (or comments module) depends on it.

## 3. Mentionable Users Endpoint

### 3.1 Contract

- **Endpoint:** `GET /api/tickets/:ticketId/mentionable-users`
- **Query params:** Optional `search` (string) — filter by name, email, or display name (e.g. case-insensitive contains).
- **Auth:** Same as ticket view — user must be able to see the ticket (existing visibility/policy). If they cannot see the ticket, return 403.
- **Response:** List of users who **have visibility to the ticket** and are **active**. Shape per item: `id`, `name`, `email`, and a display label (e.g. `displayName` or `name ?? email`) so the client can build `@[displayName](id)`. Exclude the current user if desired (optional; spec does not require excluding self).
- **Ordering:** Deterministic — e.g. by `name` asc, then `id` asc for tiebreaker.

### 3.2 Backend implementation

- **Who has ticket visibility:** Reuse Stage 1 rules. Options:
  - **Option A:** Use TicketVisibilityService to build a **where** clause for tickets, then resolve "users who can see this ticket" by: (1) load ticket (id, requesterId, ownerId, studioId, departmentId, etc.); (2) for ADMIN — all active users (or a bounded list); for DEPARTMENT_USER — users in same department(s) + owner + requester + users with studio scope; for STUDIO_USER — requester + users with that studio in scope. This is a **query over users** that matches the same logic as visibility.
  - **Option B:** Introduce a helper that returns user IDs that would pass `assertCanView` for this ticket: e.g. get all user IDs that are ADMIN, or (for this ticket) requester, owner, watchers, users in ticket's department, users with ticket's studio in scope. Then filter to active only.
- **Recommended:** Implement a **MentionableUsersService** (or method on TicketsService) that: (1) loads the ticket (minimal fields needed for visibility); (2) asserts current user can view ticket (TicketVisibilityService.assertCanView or policy); (3) runs a query that returns User records (id, name, email) for users who **can see this ticket** (same visibility logic as Stage 1, expressed as a list of user IDs or a Prisma query). For ADMIN, return active users (optionally paginated or capped, e.g. max 500). For DEPARTMENT_USER/STUDIO_USER, return only users in the same visibility "pool" (department, studio, requester, owner, watchers). Apply optional `search` filter (name/email contains).
- **Location:** New method in **TicketsService** (e.g. `getMentionableUsers(ticketId: string, actor, search?: string)`) or new **MentionableUsersService** in comments module (or shared). Controller in **CommentsController** or **TicketsController**; plan recommends **CommentsController** under same prefix: `GET tickets/:ticketId/mentionable-users` (same resource hierarchy).

### 3.3 Controller and route

- **Controller:** Add action to `CommentsController` or add a separate controller under `tickets/:ticketId`. Recommended: add `Get('mentionable-users')` in CommentsController with a route that does not conflict with `:commentId` (order routes so literal `mentionable-users` is matched before `:commentId`). Alternatively add in TicketsController as `GET /api/tickets/:ticketId/mentionable-users`.
- **Guard:** JwtAuthGuard; use CurrentUser. Call service that checks ticket visibility and returns list.

### 3.4 DTO / response

- No request body. Optional query DTO: `MentionableUsersQueryDto` with `@IsOptional() @IsString() search?: string`.
- Response: array of `{ id: string; name: string | null; email: string; displayName: string }` (or equivalent). No pagination required for Stage 3 if list is capped; otherwise add limit (e.g. 100) and optional cursor.

### 3.5 Files to touch

- New: service that implements "mentionable user IDs for ticket" (TicketsService method or MentionableUsersService). If in TicketsService, ensure no circular dependency (CommentsService → TicketsService is likely already present for ticket load in create).
- `apps/api/src/modules/comments/comments.controller.ts` — add GET `mentionable-users` (or tickets controller).
- Optional: `apps/api/src/modules/comments/dto/mentionable-users-query.dto.ts` for query validation.
- CommentsService (create flow) will use this to get allowedUserIds for mention validation (§2).

## 4. Notification Fanout Adjustments

### 4.1 COMMENT_ADDED — deduplicate with mentions

- **Current behavior:** Recipients = requester, owner, watchers; actor removed.
- **New behavior:** Same set, but **subtract** every user in `payload.mentionedUserIds` (for this comment). So: COMMENT_ADDED recipients = (requester, owner, watchers) \ mentionedUserIds. Mentioned users receive only MENTION_IN_COMMENT, not COMMENT_ADDED.
- **Implementation:** In **NotificationFanoutProcessor** (or equivalent fanout worker), when handling COMMENT_ADDED: build recipient set from requester, owner, watchers; then remove any id in `payload.mentionedUserIds`. Then remove actor. Then cap and proceed. Payload for COMMENT_ADDED must include `mentionedUserIds` (array) so the fanout processor can subtract. **CommentsService** (or domain event emitter) already has this when it emits COMMENT_ADDED; ensure payload includes `mentionedUserIds: string[]` (can be empty). Today COMMENT_ADDED payload may not include it; add it in CommentsService when emitting.

### 4.2 MENTION_IN_COMMENT — filter by ticket visibility

- **Current behavior:** Recipients = all `payload.mentionedUserIds`.
- **New behavior:** Recipients = only those in `payload.mentionedUserIds` who **have ticket visibility** for this ticket. Users without visibility must not receive a mention notification.
- **Implementation:** In the fanout processor, when handling MENTION_IN_COMMENT: (1) load ticket (already done for other events); (2) for each userId in `payload.mentionedUserIds`, determine if that user can see the ticket. Use **TicketVisibilityService** or a helper that, given a ticket and a user id, returns whether that user would pass visibility. If the worker does not have RequestUser objects, it must **load each user** and build a minimal RequestUser (id, role, departments, studioId, scopeStudioIds) then call `visibility.assertCanView(ticket, requestUser)` in a try/catch, or use a dedicated method like `userCanViewTicket(ticketId, userId): Promise<boolean>` that runs the same visibility logic. Add such a method if missing (e.g. on TicketVisibilityService: `async canUserViewTicket(prisma, ticketId, userId): Promise<boolean>` that loads ticket and user and returns true/false). Filter `mentionedUserIds` to only those for whom `canUserViewTicket` is true; use that set as recipients.
- **Payload:** MENTION_IN_COMMENT payload already has `mentionedUserIds`. No change to payload shape; only the **recipient resolution** in the processor changes (filter by visibility before adding to recipientIds).

### 4.3 Payload additions (optional)

- COMMENT_ADDED and MENTION_IN_COMMENT payloads may include **parentCommentId** (optional) when the comment is a reply, for future UI copy ("X replied to a comment"). Emit from CommentsService when creating a reply; fanout and dispatch do not need to change behavior, only pass through.

### 4.4 Idempotency

- JobId remains anchored to commentId for both COMMENT_ADDED and MENTION_IN_COMMENT. No change. Duplicate mention prevention remains: one row per (commentId, userId) in comment_mentions; fanout reads from payload (which comes from persisted mentions).

### 4.5 Files to touch

- `apps/api/src/workers/processors/notification-fanout.processor.ts` — (1) COMMENT_ADDED: subtract payload.mentionedUserIds from recipient set; ensure payload.mentionedUserIds is passed from CommentsService. (2) MENTION_IN_COMMENT: filter payload.mentionedUserIds by ticket visibility before adding to recipientIds.
- `apps/api/src/common/permissions/ticket-visibility.service.ts` — add method `canUserViewTicket(ticketId: string, userId: string): Promise<boolean>` (or similar) that loads ticket and user, builds a RequestUser-like object, and returns whether that user would pass assertCanView. This may require Prisma in the service to load user role, departments, studioId, scopeStudioIds. Alternatively implement this in the fanout processor by loading user and ticket and calling a static or injected visibility check.
- `apps/api/src/modules/comments/comments.service.ts` — when emitting COMMENT_ADDED, include `mentionedUserIds: string[]` in payload (list of userIds from CommentMention for this comment). When emitting MENTION_IN_COMMENT, payload already has mentionedUserIds (filtered by create-time validation); no change. Include `parentCommentId` in both payloads when the comment is a reply.

## 5. Comment Creation Transaction Flow

### 5.1 Create comment (top-level or reply)

- **Input:** ticketId, CreateCommentDto (body, optional parentCommentId), actor.
- **Steps (order matters):**
  1. Load ticket (for visibility and policy). If not found → 404.
  2. **Policy:** Evaluate COMMENT_ADD_PUBLIC (or equivalent). If not allowed → 403.
  3. **Reply validation (if parentCommentId present):** Load parent comment by id. If not found → 404. If parent.ticketId !== ticketId → 400. If parent.parentCommentId !== null → 400 (no reply-to-reply).
  4. **Mentionable users:** Resolve mentionable user IDs for this ticket (call MentionableUsersService or TicketsService.getMentionableUserIds).
  5. **Parse mentions:** Extract userIds from body using MentionParserService with allowedUserIds. If strict: any userId not in allowed set → 400 with clear message. If strip: only persist allowed userIds.
  6. **Transaction:** Create TicketComment (ticketId, authorId, body, isInternal, **parentCommentId**); create CommentMention rows for each (commentId, userId); update ticket.updatedAt. Commit.
  7. **After commit:** Audit log; emit COMMENT_ADDED (payload includes mentionedUserIds, parentCommentId if reply); if mentionedUserIds.length > 0, emit MENTION_IN_COMMENT (payload includes parentCommentId if reply).
  8. Return mapped comment (with author, mentions, parentCommentId, no replies array for a single comment response).
- **Double-submit:** Existing client-side disable and idempotent jobId by commentId already limit duplicate notifications; no change required in transaction.

### 5.2 Update comment (existing)

- If comment **update** re-parses body for mentions (e.g. to update CommentMention rows): run same mentionable-user resolution and parser with allowedUserIds; reject or strip invalid mentions. Update comment body and replace CommentMention set (delete existing for this commentId, insert new). Do **not** emit COMMENT_ADDED/MENTION_IN_COMMENT again for edits unless product explicitly requires "notify on edit"; spec does not require it. Plan assumes **no** notification on edit.

### 5.3 Files to touch

- `apps/api/src/modules/comments/comments.service.ts` — implement steps above; add parentCommentId to create data; validate parent; call mentionable resolution and parser; emit payloads with mentionedUserIds and parentCommentId.
- `apps/api/src/modules/comments/dto/create-comment.dto.ts` — add optional `parentCommentId?: string` with `@IsOptional()` and `@IsString()` (and optionally `@IsNotEmpty()` when present).

## 6. Comment List Query Changes

### 6.1 Where comments are loaded

- **GET /api/tickets/:id** — ticket detail includes comments. Today: flat list, orderBy createdAt asc. Change to **thread shape** (see below).
- **GET /api/tickets/:ticketId/comments** — standalone comment list. Today: flat list. Change to **thread shape** with deterministic ordering.

### 6.2 Thread shape

- **Option A (nested):** Return top-level comments only at root, each with a **replies** array. Order: top-level by `createdAt` asc (tiebreaker `id` asc); within each, replies by `createdAt` asc (tiebreaker `id` asc). So response is `comments: Array<{ ...comment, replies: Array<{ ...comment }> }>`.
- **Option B (flat):** Return flat list with `parentCommentId` on each; client groups. Spec recommends **Option A** for clarity and to avoid client grouping errors.
- **Implementation:** Query all comments for the ticket with `orderBy: [{ createdAt: 'asc' }, { id: 'asc' }]`, include author and mentions. In code, group into top-level (parentCommentId === null) and replies (parentCommentId !== null). Build tree: for each top-level, attach replies where parentCommentId === top-level.id. Sort top-level by createdAt (then id); sort each replies array by createdAt (then id). Map each node with mapCommentToResponse. Return array of top-level comments with `replies` property.

### 6.3 TicketsService.findById (ticket detail)

- **Select/include:** Add `parentCommentId` to comment select. Include relation for replies if Prisma supports (e.g. include replies with same author/mentions). Alternatively fetch comments in one query and build tree in code. Ensure **deterministic order**: `orderBy: [{ createdAt: 'asc' }, { id: 'asc' }]` for comments. If Prisma nested include is used for replies, order replies the same way.
- **Response:** Replace `ticket.comments` with the **thread-shaped** list (top-level + replies per item). So the ticket DTO or serializer returns `comments: ThreadShapeComment[]` where each has `replies: CommentResponse[]`.

### 6.4 CommentsService.findByTicket

- Same thread shape. Load all comments for ticketId with orderBy; build tree; return top-level array with replies. Each comment (and reply) mapped with mapCommentToResponse; include parentCommentId and mentions.

### 6.5 Index and query requirements

- Index on `(ticketId, parentCommentId)` is redundant if we have ticketId index and parentCommentId index; query is "where ticketId = X order by createdAt, id". Existing index on ticketId suffices; parentCommentId index helps when building replies per parent. No composite index required for Stage 3 if single-column indexes exist.
- **Deterministic ordering:** Every comment list query must use `orderBy: [{ createdAt: 'asc' }, { id: 'asc' }]` (or equivalent) so pagination and UI are stable.

### 6.6 Files to touch

- `apps/api/src/modules/tickets/tickets.service.ts` — ticket select: add parentCommentId to comments include; change comment loading to produce thread shape (top-level + replies, ordered). Update the return shape so `comments` is thread-shaped.
- `apps/api/src/modules/comments/comments.service.ts` — findByTicket: query with parentCommentId and orderBy; build tree; return thread-shaped array.
- `apps/api/src/common/serializers/comment-response.ts` — ensure mapped type includes `parentCommentId?: string | null` and optionally `replies?: CommentResponse[]` so API contract is typed. No change to mapCommentToResponse signature if it only maps a single comment; the thread builder attaches replies after mapping.

## 7. Frontend Comment Thread Rendering

### 7.1 Data shape

- **Assumption:** API returns comments as thread-shaped: `Array<{ id, author, body, createdAt, mentions, parentCommentId, replies: Array<...> }>`. Top-level items have `parentCommentId: null`; replies have `parentCommentId` set and appear in `parent.replies`.
- **Consumption:** Ticket detail page and TicketDrawer currently use `ticket.comments` as a flat array. They must be updated to consume **thread shape**: either (1) iterate top-level and for each render comment + map over `replies` with indentation, or (2) keep a flat list if API is flat and client groups by parentCommentId. Plan assumes API returns nested shape; frontend renders accordingly.

### 7.2 Components affected

- **Ticket detail page** — `apps/web/src/app/(app)/tickets/[id]/page.tsx`: section that renders `ticket.comments` (e.g. visibleComments). Change to render thread: for each top-level comment, render one block (avatar, author, body, timestamp, mentions); below it, render each reply **indented** (e.g. margin-left or nested component). Add **Reply** button/control on each top-level comment (and optionally on replies if product allows reply only to top-level; spec says one-level so Reply on top-level only, or Reply on both top-level and reply that goes to same parent — clarify: Reply creates a reply to that comment, so Reply on a reply would be reply-to-reply which is disallowed; so **Reply only on top-level comments** in UI, or Reply on replies could be disabled/hidden). So: Reply button only on top-level comments; clicking Reply opens reply composer with parentCommentId set.
- **TicketDrawer** — `apps/web/src/components/tickets/TicketDrawer.tsx`: same as above. Comments tab currently maps `ticket.comments`; change to thread rendering (top-level + indented replies). Add Reply on top-level comments; reply composer submits with parentCommentId.

### 7.3 Structural UX

- **Comment block:** Each comment (and reply) shows author, body, timestamp, mentions. Replies are **indented** (e.g. left margin or padding so visual hierarchy is clear). No nested collapse beyond one level; optional "collapse replies" when reply count > N (e.g. 5 or 10) — show "N replies" and expand/collapse control.
- **Reply composer:** When user clicks Reply, show a composer (textarea + submit) **under that comment** (or inline). On submit, POST with `{ body, parentCommentId: thatComment.id }`. Success: invalidate ticket query; refetch; new reply appears under parent.
- **Deterministic order:** Display order = API order (top-level by createdAt asc, replies by createdAt asc). No client-side re-sort.

### 7.4 Files to touch

- `apps/web/src/app/(app)/tickets/[id]/page.tsx` — comment list: thread iteration; reply button; reply composer state (which parent is being replied to); submit with parentCommentId; invalidation on success.
- `apps/web/src/components/tickets/TicketDrawer.tsx` — same: thread rendering, Reply, reply composer, parentCommentId in create payload.
- Optional: extract a **CommentThread** or **CommentList** component that accepts thread-shaped comments and renders top-level + indented replies (and optional collapse). Reduces duplication between page and drawer.

## 8. Mention Typeahead Integration

### 8.1 When user types `@`

- **Trigger:** On `@` in the comment (or reply) body, open a **suggestion dropdown** (typeahead). Input can be a contenteditable, textarea, or rich-text; plan assumes a **textarea** with cursor position or a simple "insert at end" for minimal implementation; full implementation tracks cursor and inserts token at cursor.
- **Data source:** Call **GET /api/tickets/:ticketId/mentionable-users** with optional **search** (e.g. substring after `@` until next space or end). Debounce search (e.g. 200–300 ms). Display list of users (displayName, email); on **selection**, insert **stable token** `@[Display Name](userId)` into the body at the cursor (or append if cursor not tracked).
- **Restriction:** Only users returned by mentionable-users are shown; only those can be inserted. Backend will reject if someone is mentioned who is not in the list (if strict validation).

### 8.2 Frontend components

- **Comment composer** (ticket detail and drawer): add **mention typeahead** — when body contains `@`, show dropdown; fetch mentionable-users (search = text after @); on select, insert `@[name](id)` and close dropdown. May require a wrapper around textarea that handles key events and cursor (or use a library that supports mentions).
- **Display of mentions:** When rendering comment body, optionally **linkify** or style `@[Name](userId)` (e.g. show as a pill or link to user). Do not rely on plain @ for notification; display can still show the token as-is or styled.

### 8.3 API usage

- **Initial load (optional):** On opening comment tab or focus in composer, prefetch mentionable-users (no search) to cache list; then filter client-side on `@` + typed string. Or fetch on every `@` with search = typed string (simpler, ensures fresh list).
- **Query key:** e.g. `['mentionable-users', ticketId, search]` for caching. Invalidate when ticket changes or not at all (list is permission-based and can be stale until next open).

### 8.4 Files to touch

- **Ticket detail page** and **TicketDrawer**: comment composer area — integrate mention typeahead (new component or hook). Add `useMentionableUsers(ticketId, search)` hook that calls GET mentionable-users; expose list and loading state. In composer, on `@` show dropdown, on typing filter/search, on select insert token.
- **lib/api** (or equivalent): add `mentionableUsersApi.get(ticketId, { search })` that calls GET /api/tickets/:ticketId/mentionable-users.
- Optional: shared **MentionInput** or **CommentComposer** component used by both page and drawer to avoid duplication.

## 9. Implementation Order

Implement in the following order to minimize rework and allow incremental verification.

1. **Migration and schema (Comment Thread Data Model)**  
   Add `parentCommentId`, FK, index, self-relation. Run migration. Regenerate Prisma. No behavior change yet; existing comments remain top-level.

2. **DTO and create validation (Comment Creation)**  
   Add `parentCommentId` to CreateCommentDto. In CommentsService.create, validate parent (exists, same ticket, parent.parentCommentId === null). Persist parentCommentId. Do **not** yet change list shape or notification dedupe. Verify: create top-level and create reply; GET comments still flat; replies stored correctly.

3. **Comment list query (thread shape)**  
   Update TicketsService.findById and CommentsService.findByTicket to return thread-shaped comments (top-level + replies, deterministic order). Update serializer/response type to include parentCommentId and replies. Verify: GET ticket and GET comments return nested shape; order stable.

4. **Mentionable users endpoint**  
   Implement GET mentionable-users (service + controller). Implement "users who can see this ticket" (reuse visibility logic). Add query param search. Verify: as ADMIN/DEPARTMENT_USER/STUDIO_USER, list is correct; 403 when no ticket access.

5. **Mention parsing and validation (Backend)**  
   Extend MentionParserService (or create flow in CommentsService) to accept allowedUserIds; filter or reject invalid mentions. In create (and update if applicable), resolve mentionable user IDs, call parser, reject 400 or strip. Verify: mention of user without visibility rejected or stripped; only allowed users get CommentMention rows.

6. **Notification fanout (COMMENT_ADDED dedupe, MENTION visibility)**  
   Add mentionedUserIds to COMMENT_ADDED payload; in fanout, subtract mentionedUserIds from COMMENT_ADDED recipients. Add visibility filter for MENTION_IN_COMMENT recipients (TicketVisibilityService helper or inline). Verify: owner mentioned gets one notification (mention); user without visibility not notified for mention.

7. **Frontend thread rendering**  
   Update ticket detail page and TicketDrawer to render thread (top-level + indented replies). Add Reply button and reply composer; submit with parentCommentId. Invalidate and refetch on success. Verify: replies appear under parent; order correct.

8. **Mention typeahead (Frontend)**  
   Add mentionable-users API call; on `@` in composer, show dropdown with search; on select insert `@[Display Name](userId)`. Verify: only mentionable users shown; selection inserts token; backend accepts and notifies.

9. **Payload and polish**  
   Add parentCommentId to COMMENT_ADDED and MENTION_IN_COMMENT payloads when comment is a reply. Optional: collapse/expand for long reply lists. Verify: acceptance criteria and regression (Stage 1 visibility, Stage 2 workflow unchanged).

## 10. Verification Checklist

Use this checklist to verify Stage 3 implementation without regressions.

### 10.1 Data model and API

- [ ] Migration applied; `parentCommentId` exists on `ticket_comments` with FK and index; existing rows have null.
- [ ] POST comment with `parentCommentId` (valid parent, same ticket, parent is top-level) creates reply; POST with parentCommentId pointing to another ticket’s comment returns 400; POST with parentCommentId pointing to a reply returns 400.
- [ ] GET ticket and GET comments return thread shape: top-level comments with `replies` array; each reply has `parentCommentId` set; order is deterministic (createdAt, id).
